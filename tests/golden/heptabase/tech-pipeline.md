# 完整技术链路
> [Source block]
想象一下，在一个火光冲天的工业锅炉房内，如果算法检测到了火焰异常，却因为厂房的网络波动，需要等待 2 秒钟让云端服务器返回“灭火”指令，这 2 秒钟可能就是一场灾难。在工业现场，实时性和独立性不是可选项，而是生存线。
本节我们将深入探讨这套系统的“生命周期”：数据是如何从传感器产生，经过 PC 端的智慧训练，最终在树莓派这个“独立大脑”中完成从感知到控制的惊险跳跃。

## 算力与执行的“分家”：训练部署分离模式
在嵌入式 AI 的工程实践中，最核心的共识是：不要试图在嵌入式设备上训练模型。
本项目采用了典型的“PC 端训练 + 树莓派端独立运行”模式。这种模式本质上是根据计算资源的丰瘠进行的任务分工。 [1]

### PC 端：算力中心的“育种期”
PC 或工作站配备了强大的 GPU 和充足的内存，它是模型的“摇篮”。在这里，我们使用 PyTorch 框架处理数以万计的火焰图像和振动采样。在这个阶段，我们不计代价地追求模型的准确率，并利用强大的算力进行多次迭代、超参数搜索以及混合精度训练。 [2] 

### 树莓派端：边缘终端的“战斗期”
一旦模型“训练成熟”，它就会被导出并搬家到树莓派 4B 上。此时，树莓派成为了一个独立可运行的工业智能终端。 [3] 
为什么要强调“独立”？
带宽节省：不需要将海量的原始高清图像或高频振动数据传向服务器。
极低延迟：从采集到触发继电器，整个链路在本地毫秒级闭环。
高可靠性：即使工厂网线被挖断，只要电源不断，检测系统依然能够雷打不动地进行安全监测。

## 拆解八阶段工业控制链路
一个真正能“上现场”的项目，绝不只是跑一个 model.predict()。本项目构建了一个从物理世界到数字世界，再回到物理控制的完整闭环。 [4] 我们可以将这条链路拆解为以下八个关键环节：

### 1. 采集（Data Acquisition）
这是系统感知世界的感官。对于 P1 锅炉火焰项目，是三路 USB 摄像头并行捕捉不同角度的视觉信息；对于 P2 轴承故障项目，则是通过 24-bit 高精度 ADC 模块以每秒 50,000 次的速度读取微小的振动。 [5]

### 2. 数据融合处理（Data Fusion）
工业原始数据通常充满噪声。本项目在此阶段通过 MSFD 算法 进行处理。
P1：将三路摄像头的图像进行 NCC 互相关加权融合，解决单一摄像头被遮挡或角度受限的问题。
P2：对振动信号进行 FFT（频域）和 CWT（时频域）转换。 [6]

### 3. 特征提取（Feature Extraction）
将融合后的数据转化为模型能听懂的“语言”。例如，将轴承的振动波形转化为一张 224x224 的灰度时频图。这时，原本杂乱无章的电流或振动信号，变成了具有明显视觉特征的几何图案。

### 4. 模型推理（Model Inference）
这是最硬核的一步。我们将标准化后的张量送入自研的 VariableMobileNet 模型。模型会在极短的时间内给出一个判断：这是“正常”还是“故障”？概率是多少？ [7]

### 5. 结果判断（Task Judgment）
拿到模型输出的置信度后，系统会进行逻辑判断。
防误报机制：如果模型连续 5 帧都判断为“火焰异常”，才真正触发报警。这能有效过滤掉因为光影瞬间闪烁导致的误触发。

### 6. 现场显示（Local Display）
通过 OLED 屏幕和 LED 指示灯，让现场的工人第一眼就能看到当前系统的健康状态。绿灯行、红灯停，这是最直观的人机交互。 [8]

### 7. 工业输出（Industrial Output）
这是本项目区别于普通 Demo 的核心。
物理联动：触发继电器切断电源或开启喷淋。
标准传输：通过 4-20mA 电流信号将故障严重程度传给 PLC，或通过 RS485 总线发送故障代码。 [9]

### 8. 日志记录（Logging）
最后，所有的推理结果、异常时刻的原始数据，都会被存入 SD 卡形成 CSV 日志。这为后续的事故追溯和算法改进提供了第一手资料。 [10]

## 辟谣：关于边缘 AI 的三大常见误解
在学习本项目前，我们必须厘清几个在简历和面试中容易掉进去的坑： [11]
“上位机下发决策”：很多同学认为树莓派只是个“显卡”，需要电脑告诉它怎么做。错误。 在本项目中，树莓派是 Master（主控），它自己采集、自己思考、自己执行。
“云端推理”：以为推理过程发生在远程服务器上。错误。 本项目是 100% 的边缘计算（Edge Computing），不依赖任何网络连接也能工作。
“电脑实时介入”：以为运行过程中需要连着笔记本看串口打印。错误。 电脑仅参与“训练阶段”。在现场，树莓派是一个独立封装好的铁盒子，里面只有它自己。

## ONNX：连接训练与部署的标准化桥梁
既然训练在 PyTorch（Python），部署在树莓派（C++），那么两者之间如何通信？答案是 ONNX（Open Neural Network Exchange） 格式。 [12]

### ONNX 的四个工程价值
框架解耦：部署端不需要安装沉重的 PyTorch。只要有 ONNX Runtime 这个轻量级引擎，模型就能跑。
部署通用性：同一个 ONNX 文件，你可以放在树莓派上跑，也可以未来移植到 NVIDIA Jetson 或国产国产计算卡上。
性能最大化：通过 C++ 加载 ONNX 模型，可以避开 Python 的全局解释器锁（GIL），实现真正的多线程并发并行，这在实时工业场景下至关重要。
版本管理：模型权重和结构被打包成一个 .onnx 文件，就像软件的版本号一样清晰，方便工程化维护。 [13]

### 为什么选择 C++ 而不是 Python 进行部署？
这是一个常见的面试题。虽然 Python 也能在树莓派上跑推理，但 C++ 提供了可预测的执行时间（Deterministic Timing）。在工业控制中，如果逻辑处理时而 10ms，时而 100ms，会导致控制链路的不稳定。此外，C++ 对内存的精细管理能确保系统在连续运行几个月甚至几年后，依然不会因为内存泄漏或垃圾回收（GC）导致程序卡顿或崩溃。

## 架构的基石：五层框架与核心创新
在建立了“链路”的全局认知后，你可能会好奇：这么复杂的八个步骤，代码是如何组织的？如果以后想给项目加一个“语音报警”功能，是不是要重写整个系统？
为了解决这些问题，本项目引入了五层统一架构。这种分层设计就像是搭建积木，每一层只关注自己的核心职责，通过标准化的接口与上下层通信。在这种架构下，我们还埋伏了三个重大的创新点，它们分别是解决融合问题的 MSFD 算法、解决算力问题的 VariableMobileNet，以及解决精度问题的代表性样本量化策略。这些内容，正是我们下一节要攻克的堡垒。
